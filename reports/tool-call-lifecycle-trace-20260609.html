Tool-call lifecycle trace

本 PR 搭建 tool-call lifecycle 的第一层基础设施：把每个顶层 provider tool call 记录成可串联的 event-sourced trace/ledger，但不引入 hard gate、不引入 approval policy，也不改变 provider tool_result wire。

结论先行

  • 做了什么：在 ToolExecutor 内给每个顶层工具调用生成/继承 tool_trace_id，并记录 proposal → normalized → validation → pass-through approval → dispatch → durable log visible → model visible 边界。
  • 没有做什么：没有阻断策略，没有人类 approval gate，没有改变 MCP/工具实际执行路径，没有改变 provider 可见的 tool_call_id / tool_result 对应关系。
  • 为什么现在做：为之后调试、审批基础设施、长任务恢复、模型行为分析提供稳定事件骨架；Phase 1 先只做可观测性。
  • 验证：full pytest passed — 2077 passed, 2 skipped。

实现要点

Stable trace identity

如果 provider 已给 ToolCall.id，直接用它作为 tool_trace_id；手工/测试调用无 id 时生成 tool-<uuid>。这不改变 provider-visible result id。

Sequential + parallel parity

顺序路径与 parallel-safe 路径都记录同一组生命周期边界；parallel 结果仍按原输入顺序返回。

Validation vs dispatch

unknown tool / duplicate block 不再被 trace 表达为 dispatch failure；它们在执行前停于 tool_call_validation_failed，更贴近 lifecycle 语义。

Spill visibility

oversized result spill 现在与 tool_trace_id 关联，并在 model-visible boundary 上记录 spilled=true。

测试覆盖

  • 新增 sequential success lifecycle 顺序测试。
  • 新增 parallel success lifecycle 测试，确认结果顺序不变。
  • 新增 unknown-tool validation failure 测试，确认不会出现 approval/dispatch events。
  • 新增 duplicate-block validation failure 测试，确认 blocked call 不 dispatch。
  • 新增 spilled result lifecycle 测试，确认 tool_result_spilled / durable / model visible events 共享 trace id。
  • Review follow-up: 新增 id-less trace-id fallback 与 dispatch exception lifecycle 测试。

Risk / review notes

  • Central file touched: ToolExecutor is central, so the PR keeps the change constrained to logging/trace boundaries and uses full pytest validation.
  • More logs: each tool call emits several extra structured events. Full normalized args are still present on execution-boundary events, but tool_call_received deliberately stores only raw arg keys/count to avoid durable retention of stripped hidden/control fields. If log volume becomes an issue later, we can add compaction/filtering at query/report layers.
  • No policy yet: tool_call_approved is deliberately pass-through. Real approval policy should be a later PR after the event contract has stabilized.
