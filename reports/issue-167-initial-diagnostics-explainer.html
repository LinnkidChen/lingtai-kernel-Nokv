Initial diagnostics and wire cleanup for lingtai-kernel #167

This PR is intentionally narrow: it adds evidence for the Telegram delivery gap investigation and hardens one kernel exit path. It does not claim to fix the full 15-minute Telegram delay.

Issue context

Issue #167 reports a post-turn IDLE case where a live Telegram message surfaced about 15 minutes late, then delivery recovered through a pending-tool-call heal path. The first PR should help separate three possibilities: the producer/addon wrote the event late, the kernel inbox poller failed to drain an existing event promptly, or the host/runtime was suspended or frozen.

What changed

MCP inbox boundary telemetry

mcp_inbox_event logs now include drain_latency_s, measuring how long a LICC event file sat in the kernel inbox before consumption.

If the event carries a parseable top-level received_at timestamp, the log also includes producer_lag_s for producer-to-kernel timing.

Poll-backoff wire cleanup

When the turn exits via idle_after_poll_backoff, the kernel now defensively closes any still-pending tail tool calls using the distinct reason poll_backoff_exit.

Normal adapters already commit tool results, so this is a no-op in the expected path and an attribution improvement in the abnormal path.

Doc reconciliation

The poll-backoff docstring now matches actual behavior: counters reset on sends and on check/read results that found messages. No unsupported notification-arrival reset was added.

The LICC received_at comment now describes it as an optional producer timestamp for lag telemetry.

What this intentionally does not do

  • No Telegram addon listener heartbeat or .alive supervision.
  • No auto-restart or broad listener supervision rewrite.
  • No claim that the exact observed 15-minute root cause is fixed.
  • No change to notification preview payload semantics beyond additive log telemetry.
